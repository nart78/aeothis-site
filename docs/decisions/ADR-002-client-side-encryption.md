# ADR-002: Client-Side Encryption for Proposal Pages

**Date:** 2026-03-15
**Status:** Accepted

---

## Decision

Proposal pages are encrypted client-side. The full proposal content is AES-encrypted by `encrypt_proposal.py` (in aeothis-tools) at generation time. The browser decrypts and renders the content when the visitor enters the correct access code. The static file host never sees the decrypted content.

---

## Context

Proposal pages contain commercially sensitive information: AEO scores, competitor analysis, pricing, and the client's specific strategic gaps. This content cannot be publicly accessible, but the site is hosted on GitHub Pages (a public static host).

Three access control approaches were considered:

1. **Password-protected server route** -- Requires a server. Contradicts the static hosting decision (ADR-001).
2. **Client-side encryption (current approach)** -- The file is readable only by someone with the access code. No server auth layer required. Works on any static host.
3. **Unlisted URL only** -- No encryption, just a hard-to-guess URL. Does not prevent access if the URL is shared or discovered.

Client-side encryption satisfies both constraints: the content is confidential, and the site remains fully static.

---

## Consequences

- Proposal pages are generated by the `aeodon-proposal` skill and encrypted by `encrypt_proposal.py` before committing to the repo.
- The plaintext access code is stored in `clients/{slug}/.access-code` in the aeothis-site repo (not public-linked but committed).
- The access code is included in the proposal email sent via `send_proposal_email.py`. Without it, the page is unreadable.
- If a client loses their access code, retrieve it from the `.access-code` file and resend.
- The public GitHub repo contains the encrypted HTML. The plaintext proposal content is never in the repo.
- Encryption is handled by `encrypt_proposal.py` in `~/aeothis-tools/`. The aeodon-proposal skill orchestrates the full workflow.
