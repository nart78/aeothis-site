# AEOthis Site — Architecture

## Overview

`aeothis-site` is the public-facing static website and client delivery portal for AEOthis. It serves as the marketing site, hosts per-client encrypted proposal pages and GBP guides, provides client progress dashboards, and runs the Aeori AI onboarding chatbot.

The site is a GitHub Pages deployment. It is intentionally static -- no server-side rendering, no backend. All dynamic behavior (chatbot, form submission) routes to external services.

---

## Deployment Model

- **Host:** GitHub Pages
- **Repo:** `nart78/aeothis-site` (public)
- **Domain:** `aeothis.com`
- **Deploy trigger:** Push to main branch
- **Rule:** Push after every change. No manual step needed. Do not ask for confirmation before pushing.

## Client Page Structure

Each active client gets a subdirectory under `/clients/`:

```
clients/
  {slug}/
    index.html          # Encrypted proposal page
    .access-code        # Access code file (not publicly linked)
    guide/
      index.html        # GBP guide page (generated by gbp_guide.py)
    dashboard/
      index.html        # Progress dashboard (generated by build_dashboard.py)
```

**Proposal pages** are generated by the `aeodon-proposal` skill and encrypted via `encrypt_proposal.py` in aeothis-tools. The access code is delivered to the client in their proposal email. The page is publicly accessible at the URL but content is locked behind the access code.

**Guide pages** must exist before a guide email can be sent. Generate the guide using `gbp_guide.py` in aeothis-tools, then deploy the output here before running `send_guide_email.py`.

**Dashboards** are generated by `build_dashboard.py` and deployed to the client's dashboard path. They show layer completion status and citation count.

## Aeori Onboarding Chatbot

Located at `/get-started/`. Aeori guides prospects through a short intake flow and submits their information to the webhook.

- **Form endpoint:** Formspree `xkovedvd`
- **Webhook destination:** `agents.johnnytran.ca/api/aeothis/submit`
- **Auth:** `X-AEO-Token` header (handled server-side via webhook)
- **On submission:** aeothis-tools webhook saves the payload, triggers enrichment and monitoring, and notifies Johnny via Telegram

Spam prevention is handled by Formspree (honeypot, rate limiting). No CAPTCHA on the frontend.

---

## Proposal Page Encryption

Proposal pages are encrypted HTML files. The full proposal content is encrypted client-side (AES or similar) and the correct access code decrypts and renders it in the browser.

- Encryption is performed by `encrypt_proposal.py` in aeothis-tools
- The `.access-code` file stores the plaintext code for reference
- The access code is included in the proposal email sent via `send_proposal_email.py`
- No server-side component is required for decryption -- it runs entirely in the browser

## Brand Rules

**Italic "this":** The word "this" in AEOthis is always rendered in italic in all HTML contexts.

```html
AEO<em>this</em>
```

This rule applies everywhere on the site: page titles, headings, body copy, meta descriptions, OG tags, and any other text that renders to the user.

**Colors:**
- Purple accent: `#cb6ce6`
- CTA blue: `#7ed7ed`

**CSS cache-busting:** When `style.css` is modified, update the `?v=` query parameter on all `<link>` tags that reference it. This ensures visitors get the updated stylesheet without a hard refresh.
